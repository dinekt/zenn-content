## はじめに

個人開発をやっていると、こんな悩みにぶつかりませんか。

- 企画・開発・リサーチ・マーケティング、全部一人でやらないといけない
- アイデアを思いついても「それ本当に売れるの？」と冷静に判断してくれる人がいない
- 壁打ち相手がほしいけど、深夜に付き合ってくれる人はいない

自分もずっとこの課題を抱えていました。

そんなとき、ネット上で**CLAUDE.mdに「役割」を定義してAIに部署を持たせる**というアプローチを実践している方の記事を見かけました。面白そうだったので自分でも取り入れてみたところ、これが思った以上にうまく機能しました。

ただ、運用していくうちに「AIが全体的に自分に同調しがちで、批判的な視点が足りない」という課題が出てきました。そこで**批判者ロール**を追加して、情報ソースを独立させるという工夫を入れてみたら、判断の質がかなり変わったので、そのあたりの話を共有します。

## AI仮想チームとは

### CLAUDE.mdで「役割」を定義する

Claude Codeには、プロジェクトのルートに `CLAUDE.md` というファイルを置くと、AIがそれを読み込んで振る舞いを変えてくれる機能があります。

ここに「組織構成」を書くというアイデア自体は、すでに実践されている方がいます。自分はそれを参考に、自分の個人開発に合わせた構成を組んでみました。

### ロール構成の例

こんなロールを定義しています。

```markdown
# AI仮想チーム構成

| ロール | 役割 |
|--------|------|
| 秘書 | 自分との対話窓口。TODO管理、ヒアリング記録 |
| CEO | タスクの振り分け、意思決定のログ |
| 商品企画 | 市場調査、ペインポイント発見、プロダクト企画 |
| 批判者 | 意思決定に対する批判的レビュー（独立した立場） |
| リサーチ | 技術トレンド・競合分析 |
| マーケ | コンテンツ企画、発信戦略 |
```

各ロールの詳細は、ディレクトリごとに個別の `CLAUDE.md` を置いて定義しています。たとえば商品企画なら「プロダクトアイデアはこの6軸で評価する」「『作りたいから作る』ではなく『売れるから作る』を徹底する」といったルールを書いています。

### 設定ファイルのイメージ

商品企画ロールの設定例はこんな感じです。

```markdown
# 商品企画

## ミッション
> 「作る前に、何を作るべきかを見極める」

## 評価軸（プロダクトアイデアの判断基準）
| 評価軸 | 基準 |
|--------|------|
| 痛みの深さ | 「あったらいいな」ではなく「ないと困る」レベルか |
| 市場規模 | マネタイズが現実的か |
| 競合状況 | 差別化余地があるか |
| 技術的実現性 | 個人開発で2-4週間以内にMVPが作れるか |
| AIの必然性 | AIなしでは実現困難か |
| 販売チャネル | リーチできるターゲットか |
```

CLAUDE.md に書くだけなので、コードを書く必要はありません。

## 実際にやってみた

### 商品企画ロールにアイデア出しをさせてみた

自分のSaaSプロジェクトについて「次に何を作るべきか」を商品企画ロールに考えさせてみました。市場調査から入ってペインポイントを分析し、アイデアを6つの評価軸で採点してくれます。

一人で考えていると「これいいじゃん！」で突っ走りがちですが、AIが構造的に評価してくれると、冷静に「市場規模が小さすぎるな」「競合が強すぎるな」と判断できます。

### 批判者ロールが「それ売れないよ」と止めてくれた話

個人的に一番価値を感じているのが、**批判者ロール**です。

このロールには「組織内のどの部署にも所属しない独立した立場」「データと事実に基づいて批判する」と定義しています。さらに重要なのが、**情報ソースの独立化**というルールです。

```markdown
## 情報ソースの独立性

批判者は組織内部のデータを情報ソースとして使用しない。

### 必須事項
- 独自のWeb調査で一次ソースを収集する
- 市場データ、競合情報、技術動向を自ら調べる
- 調査ソースをレビューに明記する

### 理由
内部レポートは「この方針でいこう」という前提で書かれており、
同じデータから批判しても確証バイアスの循環になる。
```

なぜこうしているかというと、商品企画が「これはいける！」と報告してきた情報を使って批判しても、結局同じ方向を向いた意見にしかならないからです。投資家が独自にデューデリジェンスするのと同じ発想ですね。

実際にあったケースとして、商品企画が「市場性が高い」と判断したアイデアに対して、批判者が独自にWeb調査を行った結果「すでに大手が参入していて競合が飽和している」ことを発見し、開発を止めたことがありました。一人で考えていたら、たぶん数週間は無駄にしていたと思います。

### 寝ている間に議論が回っていた体験

Claude Codeには `/loop` というコマンドがあります。これは指定した間隔でAIに自律的にタスクを実行させる機能です（正確には、タスクを繰り返し実行するループを作る機能）。

これを使って、**商品企画 × リサーチ × 批判者の3ロールによる探索会議**を夜中に回してみました。15分間隔で5ラウンド、合計11個のアイデアを検討させたところ、朝起きたらレポートが完成していました。

3つのロールが対立しながら議論し、最終的に全ロールが合意した結論に到達していたのは、正直驚きました。商品企画が「これがいい」と推し、批判者が「いやこの競合が問題」と返し、リサーチが追加情報を持ってくる......という流れが自動で回っていたわけです。

もちろん、その結論をそのまま採用するかどうかは自分で判断します。でも「たたき台」としての質が格段に高くなりました。

## 仕組みのポイント

### CLAUDE.md + Skills + MCP の連携

この仕組みは3つの要素で成り立っています。

- **CLAUDE.md**: ロールの定義、行動ルール、評価基準を記述する設定ファイル
- **Custom Skills（カスタムスキル）**: スラッシュコマンドで呼び出せる定型業務の自動化。自分は9本ほど作っています（セッションログ記録、プロダクトブリーフィング作成など）
- **MCP（Model Context Protocol）**: 外部ツールとの連携プロトコル。Web検索、ブラウザ操作、ナレッジグラフへの記憶保存などを接続しています

特にMCPでWeb検索やスクレイピングを繋いでいるおかげで、批判者が独自に市場調査を行えるようになっています。

### 批判者の「情報ソース独立化」

繰り返しになりますが、この工夫が一番効いています。

AIは基本的に「求められた方向に合わせる」傾向があります（同調バイアス / sycophancy と呼ばれます）。商品企画が書いた楽観的なレポートを読ませてから「批判して」と言っても、表面的な指摘しか出てきません。

そこで、批判者には**内部レポートを参照させず、Web検索で独自に情報を集めさせる**ルールにしました。こうすると、内部では見落としていた競合情報やリスクが出てくるようになります。

### セッションの流れを仕組み化する

毎回のセッション（Claude Codeを起動してから終了するまで）にも決まったフローを設定しています。

1. 前回の振り返り
2. 「今日何をやるか」のヒアリング
3. タスク確認
4. プロダクトのブリーフィング
5. 方針決定

こうすることで、セッションのたびにゼロから説明する手間がなくなります。AIが前回の続きから始めてくれる感覚です。

## やってみて気づいたこと

### 良かった点

**壁打ち相手が24時間いる感覚**が一番大きいです。深夜にふとアイデアを思いついたとき、すぐに商品企画に市場性を聞いて、批判者にリスクを洗い出してもらえます。

また、**判断の記録が自然と残る**のも良い点です。CEOロールが意思決定のたびにログを残すルールにしているので、「なぜこの判断をしたのか」が後から追えます。

### 注意点

一方で、**AIに任せすぎると判断がブレる**という罠もあります。AIが出した結論をそのまま受け入れてしまうと、自分の軸がなくなります。

大事なのは、**最終判断は必ず人間がする**ということ。AIチームはあくまで「情報を整理してたたき台を作ってくれる存在」であって、意思決定者ではありません。自分の場合は、AIが出したレポートを読んだうえで「よし、これでいこう」「いや、こっちの方がいい」と自分で決めるようにしています。

もうひとつ、**設定を作り込みすぎない**ことも大事です。最初は完璧なCLAUDE.mdを作ろうとして時間を使いすぎました。まずは最小限のロール定義から始めて、使いながら育てていくのが良いと思います。

### 「一人だけど一人じゃない」開発スタイル

これが一番しっくりくる表現です。コードを書くのは自分ですし、最終判断も自分です。でも「何を作るべきか」「このアイデアに穴はないか」「市場はどうなっているか」を一緒に考えてくれるチームがいる。

チーム開発の良い部分（多角的な視点、議論による精度向上）を、個人開発でも享受できる。そんな感覚です。

## おわりに

やっていることは「CLAUDE.mdに役割を書く」というシンプルなことですが、これだけで個人開発の体験がかなり変わりました。

特に、**批判者ロールの情報ソース独立化**は、AIの同調バイアスを構造的に解決するアプローチとして、他の場面でも応用できると思います。

AI仮想チームという考え方自体は自分のオリジナルではなく、先にやっている方々がいます。ただ、その中で**批判者ロールの独立性を担保する仕組み**は、自分なりに試行錯誤して辿り着いたものです。

CLAUDE.mdに数行書くだけで始められるので、興味があればぜひ試してみてください。最初は秘書ロールと批判者ロールの2つだけでも、十分に効果を実感できるはずです。
